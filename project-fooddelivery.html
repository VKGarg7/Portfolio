PROJECT 02 · INSPECTION REPORT

Food Delivery

A full-stack food delivery app with real-time menu updates, secure checkout, and role-based access for three different user types.

CLASSWeb Application
ENGINEReact / Redux Toolkit / Node.js
SYSTEMSExpress / MongoDB / Stripe
STATUSDEPLOYED
GitHub Live Demo
01

Overview

Food Delivery is a full-stack ordering platform that connects three different kinds of users on one backend: customers browsing menus and placing orders, restaurants managing what they sell, and delivery agents fulfilling orders. Rather than being a single-role storefront, the app had to model three overlapping workflows without letting any one role see or touch data it shouldn't.

02

The Challenge

The core difficulty wasn't the UI — it was the access model. A customer, a restaurant, and a delivery agent all interact with the same order object, but each should only be able to see and modify the parts relevant to their role: a customer places an order and tracks it, a restaurant accepts/prepares it, a delivery agent picks it up and marks it delivered. Getting this wrong in either direction — too permissive or too restrictive — breaks the app for one of the three user types.

03

The Engineering

Role-based access control is enforced at the API layer: each Express route checks the authenticated user's role before allowing an action on an order, rather than trusting the frontend to only show the "right" buttons to each user type. Redux Toolkit manages client-side state for menus, cart, and order status so the UI can reflect live order-status changes (placed → accepted → out for delivery → delivered) without a full page reload. Checkout is handled through Stripe, keeping payment logic out of the app's own database layer.

04

06

Key Features

  • Real-time menu updates

    Restaurants can update menu items and availability reflected immediately to customers.

  • Three-role access system

    Customer, restaurant, and delivery agent each get a scoped view of the same order data.

  • Secure Stripe checkout

    Payment processing handled through Stripe rather than a custom implementation.

  • Order status lifecycle

    Orders move through placed, accepted, out-for-delivery, and delivered states.

07

Performance

No formal load-testing or production traffic data exists for this project — it hasn't handled real restaurant order volume. The performance-relevant design choice is architectural: state transitions for an order are driven by backend role checks rather than frontend assumptions, so the correctness of "who can do what" doesn't degrade as usage grows, even though throughput hasn't been benchmarked.

08

Results

The application is deployed and live (linked above) with all three user roles functional against a shared backend — a customer can place a real order, a restaurant can manage it, and a delivery agent can fulfill it, all through the same deployed system rather than three disconnected prototypes.

09

Lessons

Designing for three roles from the start forced a clearer separation between "what the UI shows" and "what the API allows" — early on it would have been easy to just hide buttons in the frontend for the wrong role and call it done, but that's not real access control. Pushing the permission logic into the backend routes made the system actually secure rather than just visually tidy, which was the most transferable lesson from this build.
