FinPilot AI

FinPilot AI
01

Overview

FinPilot AI is an early-stage personal financial intelligence platform that aggregates accounts, cards, investments, loans, and insurance into one place, then layers an AI coaching assistant on top of that aggregated view. The live link points to a /prototype build used for UX validation — this project is honestly scoped as scaffolding and architecture in progress, not a finished production app.

02

The Challenge

The central risk in any "AI financial coach" product is hallucination: an LLM confidently stating a wrong balance, a wrong interest calculation, or made-up advice based on incomplete context is far worse than the assistant simply not answering. The system needed a way to let an LLM reason conversationally about a user's finances while guaranteeing that every number it states is pulled from a verified, deterministic calculation rather than generated by the model itself.

03

The Engineering

The architecture deliberately splits responsibilities across services instead of putting everything behind one LLM call. A Java/Spring Boot 3 service owns deterministic financial computation — balances, aggregation, loan/insurance data — backed by PostgreSQL. A separate Python/FastAPI service handles the AI reasoning layer, orchestrating multi-step reasoning with LangGraph and using Qdrant for vector retrieval over financial context, with OpenAI/Gemini as the underlying model providers. The AI layer queries the deterministic layer for facts rather than inventing them, so its output is grounded in real computed data.

Infrastructure-wise, the project is set up with Docker, RabbitMQ for async messaging between services, and MinIO for object storage, with GitHub Actions scaffolded for CI — reflecting a multi-service system designed to scale past a single monolith, even though it's still early.

Key Features

  • AI coaching layer

07

Performance

This project is intentionally not benchmarked yet — it's in the prototype/UX-validation stage, not production. The architectural decision that most affects future performance is the separation of deterministic computation from AI reasoning: the compute-heavy financial calculations don't need to round-trip through an LLM, which keeps the expensive AI calls scoped to reasoning rather than arithmetic.

08

Results

A live prototype build is deployed (linked above) for UX validation of the coaching flow and financial dashboard UI. The full multi-service backend — Spring Boot, FastAPI/LangGraph, RabbitMQ, MinIO — exists as scaffolding and is still being built out; this project is presented honestly as early-stage rather than as a finished product.

09

Lessons

Designing the AI/deterministic split before writing much code forced an early answer to "what can this system say for certain vs. what is it guessing" — a distinction that's easy to blur once an LLM is in the loop and hard to retrofit afterward. Scoping this project publicly as a prototype rather than overstating its completeness was also a deliberate choice, matching the same "honest about scope" approach used across the other project write-ups here.
