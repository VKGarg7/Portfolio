MovieTix

A full-stack movie ticket booking platform with real-time seat selection and secure payments.

01

Overview

MovieTix is a full-stack movie ticket booking platform that lets a user browse showtimes, pick seats on a live seat map, and pay securely — the same flow as a real cinema booking site, built end-to-end. It handles authentication, seat inventory, payment processing, and booking confirmation as one connected system rather than a static UI mockup.

02

The Challenge

Seat booking is a classic concurrency problem: two users can try to book the same seat for the same showtime within milliseconds of each other. A naive implementation checks seat availability, then writes the booking — but between those two steps, another request can slip in and double-book the seat. The system needed to guarantee that once a seat is marked as booked, no other request could ever claim it, without making the checkout flow feel slow or fragile to the user.

03

The Engineering

The booking flow is built around treating seat selection as a short-lived lock rather than an instant write. When a user selects seats, the backend marks them as held for that session and showtime, before payment even starts. Stripe handles the actual payment; a webhook listener then confirms the booking only after payment succeeds, and releases the hold if the session expires or payment fails. This means a seat can't be paid for by two people, and a seat also can't stay "stuck" as held forever if someone abandons checkout.

Authentication is handled by Clerk rather than a hand-rolled auth system, which reduced the surface area for security mistakes around session handling and let the engineering effort go into the booking logic itself — the part of the app that's actually novel.

06

Key Features

  • Real-time seat selection

    Live seat map that reflects seats already held or booked by other users.

  • Secure Stripe checkout

    Payment handled entirely through Stripe rather than custom card handling.

  • Clerk authentication

    User sign-in and session management delegated to a dedicated auth provider.

  • Webhook-driven confirmation

    Bookings are only finalized after Stripe confirms payment succeeded, not before.

  • Admin dashboard control

    Admin-side visibility into bookings and showtimes.

07

Performance

No formal load-testing or production traffic metrics have been collected for this project yet — it hasn't been run at real cinema-booking scale. What can be said with confidence is architectural: the seat-hold-then-confirm pattern is designed to stay correct under concurrent requests, since seat state changes happen as atomic writes rather than read-then-write from the client. The honest performance story here is "designed for correctness under concurrency," not a benchmarked throughput number.

08

Results

The project is deployed and publicly live (linked above), with the full booking flow — browse, select seats, pay, receive confirmation — working end-to-end rather than stopping at a UI prototype. That's the actual deliverable: a working payment-integrated booking system, not just a seat-map mockup.

09

Lessons

Building the seat-hold logic made the gap between "looks correct" and "is correct under concurrency" concrete rather than theoretical — a UI that visually prevents double-booking in a single-user test can still fail the moment two people click at once, so the guarantee has to live in the backend's write logic, not the frontend. Delegating auth to Clerk instead of building it from scratch was also a deliberate scope decision: it kept the engineering focus on the booking logic that actually made this project worth building.
